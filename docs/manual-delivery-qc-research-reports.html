Quality checks: Research reports

Author

Jason Wallin, OEDA-GIS Manager

Document structure and output

  1. Quarto uses embed-resources: true in the yaml heading so that the document is a single file with no external dependencies
  2. Quarto compiles without errors and warnings
  3. All outputs render: no broken figures, tables
  4. All citations and bibliography entries appear without breaking, are formatted correctly, and links function as expected
  5. All internal links and references (e.g. Table X, Figure Y, Section Z) work and point to the right content

Statistical and analytical integrity

  1. All text has been run through the statcheck package, to determine if reported p-values, df, and test statistics align. Any errors or decision errors have been resolved.
  2. All tests have effect sizes reported.
  3. Confidence intervals are included around point estimates.
  4. Data visualizations and text/data/statistics match (e.g. means in the text match the means in the visualization)

Reproducibility

  1. Quarto project structure follows OEDA standards (TBD: e.g., report.qmd, figures/, data/, scripts/) for transparency.
  2. Document makes use of a reproducible environment (TBD: determine the OEDA standard: Docker, renv.lock, etc.?)
  3. Seeds are set explicitly for simulations, resampling, bootstrapping, or other random processes
  4. Scripts and data used in the document are versioned and reproducible
  5. Code chunks use #| eval: true unless intentionally disabled with a comment as to why
  6. Input data is described or linked clearly (e.g. source, format, preprocessing). Data are all versioned instances (e.g. a flatfile, csv, Parquet, etc.), not SQL pulls from “live” administrative data.

Interpretibility and clarity

  1. Title reflects key findings, variables, populations, and time periods
  2. Abstract reflects key findings, variables, populations, and time periods
  3. Key findings are clearly stated in plain language
  4. The main message is consistent across the title, abstract, results, visualizations, and conclusion
  5. Effects are described in units meaningful to the audience (counts of individuals, dollars, days in care, etc.)
  6. Acronyms and technical terms are minimal, and are defined at first use
  7. Assumptions and limitations are acknowledged as appropriate in introduction, methods, and conclusions, and are addressed in suggestions for future work
  8. Figures and tables are interpretable without reference to the text

Writing quality style

  1. No placeholder text (ADD, TODO, etc.) remains
  2. Grammar, spelling, and punctuation are correct
    • gramr provides a writing style linter
    • ChatGPT Grammar Checker might be an option; untested at time of writing
  3. Voice and tense are consistent
  4. Terminology and variable names are consistent throughout
  5. Readability scores have been calculated and are within OEDA adopted thresholds

If you’ve rendered your .qmd to .html, here is a workflow that can help you measure readability scores:

library(rvest)
Warning: package 'rvest' was built under R version 4.4.3
library(sylcount)

# You can use a local path or a web-based path. I'll use a Wikipedia page on Oregon as an example
doc_path <- "https://en.wikipedia.org/wiki/Oregon"

# Identify the elements that contain the major body text in your document
# Usually these are <p> HTML elements
# If you want to ignore headings in your readability stats, remove the h1, h2, etc. 
element_set <- "p, h1, h2, h3, h4"

# Add any additional container elements in your document. For instance, if you've incorporated text-heavy tables, you might add td to this set

page <- rvest::read_html(doc_path)
elements <- rvest::html_elements(page, element_set)
text <- rvest::html_text(elements) |> 
  paste(collapse = '\n')

sylcount::readability(text) |> 
  knitr::kable()

Ethical, legal, and confidentiality checks

  1. No confidential or identifying information is used in any data input
  2. No confidential or identifying information appears in any output (text, visualization, table, etc.)
  3. Appropriate masking is used for all static elements
  4. Appropriate masking has been tested for with any dynamic elements
  5. All data use complies with licenses, access conditions, and consent
  6. Ethics approval, DUA, IAA, etc. are acknowledged if applicable
